Introduction

Now that instances of EmbeddedContent and SpecificResource (at least) can have role / motivation (at least in some circumstances), the question has arisen as to whether an Annotation-level motivatedBy property is still useful. This position paper proposes that motivatedBy at the Annotation level is still needed to support certain common, intuitive and compelling use cases, enumerated and illustrated in Section 3 below. The consensus of the Editors of this position paper is that these use cases provide sufficient reason to retain Annotation-level motivatedBy.

Benefits and concerns specific to each example have been described below for some illustrations. A general drawback to retaining Annotation-level motivation common to all use cases enumerated here will be the need to look for motivation / role information on the Annotation as well as on SpecificResource and EmbeddedContent. This undoubtedly will have some performance impact for processing large-scale aggregations of annotations. Since this concern is common to all use cases it is not articulated seperately for each use case below.

A second group of possible other uses of motivatedBy at the Annotation-level are enumerated and illustrated in Section 4 below, with the distinction and caveat being that these potential uses may be semantically unsound and/or semantically inconsistent with the other use cases identified. The editors did not reach consensus regarding the soundness or consistency of the potential use cases illustrated in Section 4. These use cases are not proposed or recommended, but they are included for completeness in case of interest to others.

Definition of Annotation-level motivatedBy in the Current Model

The December 2014 First Public Working Draft (FPWD) of the Web Annotation Data Model [annotation-model] defines the Annotation-level motivatedBy property as follows:
"In many cases it is important to understand the reasons why the Annotation was created, not just the agents involved.... Each Annotation should have at least one oa:motivatedBy relationship to an instance of oa:Motivation, which is a subClass of skos:Concept." An ontology of Motivations is provided in the FPWD, e.g., oa:commenting - The motivation that represents a commentary about or review of the Target resource(s). For example to provide a commentary about a particular PDF.

If retained, the definition of motivatedBy given in the Data Model will need to be revised to reflect its modified semantics, to align / differentiate its use as compared to the same or similar property as used on EmbeddedText and SpecificResource and to clarify scenarios when it is not applicable. A determination will need to be made whether potential values for the Annotation-level motivatedBy property are coincident with, overlapping with, or distinct from values appropriate for expressing the role / motivation of EmbeddedTexts and SpecificResources used as Annotation bodies and targets. Further discussion is required before a new definition can be written and ontological constraints decided.

Compelling Uses of Annotation-level motivatedBy

Expressing Motivation of the Annotation as a Whole

Expressing the motivation for the annotation as a whole is distinct from expressing the role played by bodies and targets within the annotation. motivatedBy on the Annotation is an *additional* feature to the uses of hasRole on bodies and targets, and is actually orthogonal to the use of hasRole on a Body or Target resource. The two properties can be applied independently, have different semantic meanings and may both appear within the same annotation. In such cases they can provide complementary information. In this illustration the annotation as a whole is motivated by reviewing (new Motivation) while the role / motivation of each body is different, in this case more granular, e.g., commenting and a describing (as opposed to something else that might fall within scope of reviewing).

  • JSON-LD
  • Turtle
{
    "@context": "http://www.w3.org/ns/anno.jsonld",
    "id": "http://example.org/anno1",
    "motivation": "reviewing",
    "type": "Annotation",
    "target": "http://example.org/target1" ,
    "body": [
        {
            "role": "commenting",
            "content": "http://example.org/body1"
        },
        {
            "role": "describing",
            "content": "http://example.org/body2"
        }
    ]
}



<http://example.org/anno1> a oa:Annotation ;
    oa:motivatedBy oa:review ;
    oa:hasTarget <http://example.org/target1> ;
    oa:hasBody [
        oa:hasRole oa:commenting ;
        oa:hasSource <http://example.org/body1> ],
       [oa:hasRole oa:commenting ;
        oa:hasSource <http://example.org/body1> ] .

Benefits: Provides flexibility and expressiveness; the annotator can express motivation of the annotation only, the role/motivation of the body/target only, or both.

Concerns: To the extent that Annotation-level motivation relates to body/target role/motivation it is possible to make assertions that are contradictory.

Expressing Motivation in the Absence of a Body

Bodies are not required. In the absence of a Body, an Annotation-level motivation becomes the only way (within our data model) to convey the intent of the annotation if that motivation is not suitable to attach to the hasTarget SpecificResource.

  • JSON-LD
  • Turtle
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "http://example.org/anno1",
  "type": "Annotation",
  "target": "http://example.org/target1" ,
  "motivation": "bookmarking"
}

<http://example.org/anno1> a oa:Annotation ;
    oa:hasTarget <http://example.org/target1> ;
    oa:motivatedBy oa:bookmarking .

Benefits: Preserves ability to express motivations in absence of a Body.

Concerns: May not be needed if a suitable motivation can always be assigned to the Target in all use cases lacking a Body.

Expressing Motivation for a Single Body, Simple Textual Body Annotation

In the case of Single Body, Simple Textual Body Annotations, expressing the motivation of the Annotation as a whole avoids the need to resort to Embedded Textual Body in order to express motivation / role.

  • JSON-LD
  • Turtle
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "http://example.org/anno1",
  "type": "Annotation",
  "motivation": "commenting" ,
  "target": "http://example.org/target1" ,
  "body": "A literal string comment."
}

<http://example.org/anno1> a oa:Annotation ;
  oa:hasTarget <http://example.org/target1> ;
  oa:motivatedBy oa:commenting ;
  oa:hasBody "A literal string comment" .

Benefits: The single literal body (Simple Textual Body) annotation is a common use case; this keeps the modeling and serialization extremely lightweight while still allowing the annotator to express motivation.

Concerns: Expressing a motivation for the annotation as a whole is semantically not the same as expressing role/motivation for a Body; implementers are likely to gloss over this distinction (as they do now in absence of any way to associate a role with a Body using the current data model).

Other Possible Uses of Annotation-level motivatedBy

Inherited Motivation

A use case identified but considered potentially unsound is to use Annotation-level motivatedBy as a simple way (from serialization perspective) to express Body role / motivation when it is the same for all Bodies in a multi-body annotation. This avoids having to turn each Body into a SpecificResource or EmbeddedContent. In essence (though not really since RDF doesn't work that way even if JSON sometimes does) the motivatedBy property is inherited by Bodies (and Targets?) from the Annotation.

:

  • JSON-LD
  • Turtle
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "http://example.org/anno1",
  "type": "Annotation",
  "motivation": "commenting" ,
  "target": "http://example.org/target1" ,
  "body": [ 
       "Literal string comment 1.", 
       "Literal string comment 2.", 
       "Literal string comment 3."
  ]
}

<http://example.org/anno1> a oa:Annotation ;
  oa:hasTarget <http://example.org/target1> ;
  oa:motivatedBy oa:commenting ;
  oa:hasBody "Literal string comment 1" ;
  oa:hasBody "Literal string comment 3" ;
  oa:hasBody "Literal string comment 3" .
And non-literal Bodies:
  • JSON-LD
  • Turtle
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "http://example.org/anno1",
  "type": "Annotation",
  "motivation": "describing" ,
  "target": "http://example.org/target1" ,
  "body": [ 
      {"id": "http://example.org/body1"}, 
      {"id": "http://example.org/body1"}, 
      {"id": "http://example.org/body1"}
   ]
}

<http://example.org/anno1> a oa:Annotation ;
  oa:hasTarget <http://example.org/target1> ;
  oa:motivatedBy oa:commenting ;
  oa:hasBody <http://example.org/body1> ;
  oa:hasBody <http://example.org/body2> ;
  oa:hasBody <http://example.org/body3> .

Default motivation

The use case of having an Annotation-scoped default motivation was identified but seen as semantically different than other use cases identified; if found compelling it may be that a different property would be required. Alternatively there may be better ways to accomplish the same end. This use case leverages Annotation-level motivatedBy as a way to express the default role / motivation of any Body (and Target?) in the Annotation not having an explicit role / motivation. This reduces verbosity. The role of bodies 1 and 3 is commenting.

:

  • JSON-LD
  • Turtle
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "http://example.org/anno1",
  "type": "Annotation",
  "motivation": "commenting" ,
  "target": "http://example.org/target1" ,
  "body": [ 
       "Literal string comment (body 1).",
       { "role": "questioning",
         "text": "I have a question about this example (body 2)."},
       { "id": "http://example.org/body3"}, 
       { "role": "describing",
         "content": "http://example.org/body4" }
  ]
}

<http://example.org/anno1> a oa:Annotation ;
  oa:hasTarget <http://example.org/target1> ;
  oa:motivatedBy oa:commenting ;
  oa:hasBody "Literal string comment 1" ;
  oa:hasBody [ oa:hasRole oa:questioning ;
               oa:hasText "I have a question about this example (body 2)."] ;
  oa:hasBody  <http://example.org/body3>;
  oa:hasBody [ oa:hasRole oa:questioning ;
               oa:hasContent <http://example.org/body4>] .

Fallback motivation

The use case of having an Annotation-scoped fallback motivation was identified but seen as semantically different than other use cases identified; if found compelling it may be that a different property would be required. Alternatively there may be better ways to accomplish the same end. This use case leverages Annotation-level motivatedBy as a way to express the fallback role / motivation of any Body or Target in the Annotation having an explicit role / motivation in a non-oa namespace and not recognized by an agent consuming the Annotation.

:

  • JSON-LD
  • Turtle
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "http://example.org/anno1",
  "type": "Annotation",
  "motivation": "commenting" ,
  "target": "http://example.org/target1" ,
  "body": [ 
       { "role": "ex:copyEditorRemark",
         "text": "I have a question about this example (body 1)."} ,
       { "role": "ex:seriesEditorReview",
         "content": "http://example.org/body2" }
  ]
}

<http://example.org/anno1> a oa:Annotation ;
  oa:hasTarget <http://example.org/target1> ;
  oa:motivatedBy oa:commenting ;
  oa:hasBody [ oa:hasRole ex:copyEditorRemark ;
               oa:hasText "I have a question about this example (body 2)."] ;
  oa:hasBody [ oa:hasRole ex:seriesEditorReview ;
               oa:hasContent <http://example.org/body2>] .
